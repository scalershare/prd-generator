### 1.3 有状态 vs 无状态标注

| 组件 | 状态类型 | 水平扩展难度 | 说明 |
|------|---------|------------|------|
| Web应用 | 无状态 | 容易 | 可直接加实例 |
| PostgreSQL | 有状态 | 需主从 | 数据在磁盘上 |
| Redis | 有状态 | 需集群 | 缓存可丢，队列不可丢 |
| 文件存储 | 有状态 | 需对象存储 | 本地磁盘无法共享 |

### 1.4 服务依赖关系与启动顺序

```
启动顺序：PostgreSQL → Redis → Web应用 → Worker → Nginx

依赖健康检查：
- Web应用启动前：等待 PostgreSQL 端口可达 + 可执行查询
- Worker启动前：等待 Redis 端口可达 + 可执行 PING
- 如果依赖未就绪：重试N次，间隔M秒，超过后退出（而非无限等待）
```

## 二、RTO/RPO 定义

| 指标 | 含义 | 参考值（按项目规模选） |
|------|------|---------------------|
| **RTO** | 系统宕机后多久内必须恢复 | 内部工具：4小时；面向公众：1小时；交易系统：15分钟 |
| **RPO** | 最多能接受丢失多长时间的数据 | 低风险：24小时（日备份）；中风险：1小时；高风险：0（实时复制） |

**RTO决定**：是否需要自动重启（Docker restart: always）、热备、负载均衡
**RPO决定**：备份频率、是否需要WAL归档/实时复制

## 三、熔断器与自动恢复

### 3.1 熔断器模式（Circuit Breaker）

适用于：所有外部API调用（AI服务、支付、短信、第三方数据接口）

```
状态机：
CLOSED（正常）→ 连续N次失败 → OPEN（熔断）
OPEN（熔断）→ 等待M秒 → HALF-OPEN（试探）
HALF-OPEN → 成功 → CLOSED（恢复）
HALF-OPEN → 失败 → OPEN（继续熔断）

推荐参数：
- 触发熔断：连续5次失败 或 30秒内失败率>50%
- 熔断持续：30秒
- 试探请求：1次
```

**Python实现推荐**：`circuitbreaker` 库 或 自行实现装饰器
**前端表现**：熔断期间直接走降级方案，不让用户等待

### 3.2 自动恢复机制

| 故障类型 | 自动恢复方案 |
|---------|------------|
| 服务进程崩溃 | Docker `restart: always` + healthcheck |
| 数据库连接断开 | 连接池自动重连（SQLAlchemy pool_pre_ping=True） |
| Redis连接断开 | 客户端自动重连 + 缓存穿透保护 |
| 外部API超时 | 熔断器 + 指数退避重试 |
| 磁盘空间不足 | 告警阈值80% + 自动清理临时文件/旧日志 |

### 3.3 重试策略标准

| 场景 | 重试次数 | 间隔策略 | 超时后行为 |
|------|---------|---------|-----------|
| AI API调用 | 3次 | 指数退避(1s,2s,4s) | 走降级方案 |
| 数据库写入 | 2次 | 固定间隔(500ms) | 报错+记录日志 |
| 支付回调确认 | 5次 | 指数退避(1s,2s,4s,8s,16s) | 人工对账队列 |
| 短信/邮件发送 | 3次 | 固定间隔(2s) | 记录失败+后续补发 |

**关键原则**：只有幂等操作才能重试。非幂等操作（如创建订单）重试前必须先查询是否已成功。

## 四、优雅降级层级

定义系统在不同故障程度下的行为：

| 级别 | 触发条件 | 用户可用功能 | 用户看到的提示 |
|------|---------|------------|--------------|
| **L0 正常** | 所有服务正常 | 全部功能 | 无 |
| **L1 部分降级** | AI服务不可用 | 基础功能正常，AI功能展示缓存结果或"暂时不可用" | "AI服务繁忙，请稍后重试" |
| **L2 核心降级** | 数据库只读（磁盘满/主从切换） | 只能查看，不能写入 | "系统维护中，暂时无法提交" |
| **L3 紧急降级** | 主服务不可用 | 静态维护页面 | "系统正在紧急维护，预计X点恢复" |
| **L4 完全不可用** | 服务器/域名不可达 | 无 | 需通过其他渠道通知用户（公告/群消息） |

每个级别必须定义：**触发条件**、**自动/手动切换方式**、**用户看到什么**、**运维需要做什么**。

## 五、跨系统状态一致性

适用于：涉及支付、外部下单、第三方数据同步的场景。

### 5.1 三层保障机制

```
第一层：回调确认
  - 接收第三方的异步回调通知（如微信支付回调）
  - 回调接口必须幂等（同一通知重复收到不重复处理）

第二层：主动查询
  - 超时未收到回调时，主动查询第三方接口确认状态
  - 查询策略：支付发起后30秒、1分钟、3分钟各查一次

第三层：定时对账
  - 每日/每小时拉取第三方交易记录，与本地订单比对
  - 不一致的订单进入人工审核队列
```

### 5.2 订单状态机设计原则

```
错误做法：待支付 → 已取消（支付超时直接取消）
正确做法：待支付 → 支付中 → 已支付/支付失败/支付超时待确认

"支付超时待确认"状态：
- 不释放资源（不退票/不释放库存）
- 触发主动查询第三方
- 查询确认失败后才流转到"已取消"并释放资源
```

## 六、超时分级策略

不同类型的操作，超时阈值和用户体验不同：

| 操作类型 | 超时阈值 | 用户体验 | 超时后行为 |
|---------|---------|---------|-----------|
| 普通API请求 | 3秒 | 加载动画 | 报错提示"网络异常，请重试" |
| AI生成/复杂计算 | 30秒 | 进度条+预估时间 | 提示"生成超时，已记录，稍后自动重试" |
| 文件上传 | 60秒 | 上传进度条 | 提示"上传失败，请检查网络后重试" |
| 批量导出 | 5分钟 | 转后台任务 | 邮件/站内信通知下载链接 |
| 支付等待 | 5分钟 | 倒计时+可取消 | 进入"待确认"状态 |

## 七、默认值/Fallback清单

每个依赖外部数据的功能点，必须定义"拿不到数据时用什么"：

| 数据/服务 | 正常来源 | Fallback方案 | Fallback的用户感知 |
|----------|---------|-------------|-------------------|
| AI推荐结果 | AI API实时生成 | 展示上次缓存结果 | 标注"基于上次生成" |
| 用户头像 | 用户上传 | 默认头像（首字母） | 无感知 |
| 配置参数 | 数据库/配置中心 | 代码中的硬编码默认值 | 无感知 |
| 外部数据接口 | 第三方API | 本地缓存副本 | 标注"数据更新于X时间" |

## 八、前端弱网/离线兜底

| 场景 | 兜底方案 |
|------|---------|
| 填写长表单时网络断开 | localStorage自动暂存，恢复后提示"有未提交的草稿" |
| 提交请求超时 | 自动重试1次，仍失败则提示"已保存草稿，请检查网络后重试" |
| 弱网环境（高延迟） | 关键交互加loading状态，防止重复点击 |
| WebSocket断连 | 自动重连+重连期间提示"连接中..." |

## 九、人工应急预案（Manual Fallback）

**当技术降级也失败时，最后的安全网是人。**

### 9.1 应急响应团队

| 角色 | 责任 | 联系方式 |
|------|------|---------|
| 值班运维 | 第一响应，尝试技术恢复 | （填写） |
| 技术负责人 | 值班运维30分钟内未解决时升级 | （填写） |
| 业务负责人 | 决定是否启动人工兜底流程 | （填写） |
| 第三方服务商 | 支付/云服务/AI服务的技术支持 | （填写紧急工单/电话） |

### 9.2 人工兜底流程

| 场景 | 人工兜底方案 |
|------|------------|
| 在线支付完全不可用 | 线下扫码转账+人工录入订单 |
| 系统完全不可用 | 纸质/Excel记录，系统恢复后补录 |
| 数据错乱需排查 | 逐笔人工对账+退款/补发 |

## 十、事故分级与响应标准

| 级别 | 定义 | 响应时间 | 恢复目标 |
|------|------|---------|---------|
| **P0** | 全站不可用 / 数据丢失 / 安全事件 | 15分钟内响应 | 1小时内恢复 |
| **P1** | 核心功能不可用（支付/登录/核心业务） | 30分钟内响应 | 4小时内恢复 |
| **P2** | 非核心功能异常（统计/导出/通知） | 2小时内响应 | 24小时内恢复 |
| **P3** | 体验问题（UI错位/文案错误） | 下个工作日 | 下个版本修复 |
