# PRD 文档章节模板（v4.0）

## 必须包含的章节结构

```
1. 项目概述与目标
   1.1 项目背景（为什么做）
   1.2 V1核心目标（一句话定位 + 量化指标）
   1.3 目标用户（角色表：角色名、role字段值、特征、核心需求）
   1.4 架构设计要点（多租户/单租户、隔离策略）

2. 技术架构
   2.1 技术栈（前端/后端/AI/部署，每项标注具体版本号）
   2.2 技术选型理由（为什么选这个方案，简要说明）
   2.3 系统架构图（ASCII图，标注组件职责和交互）
   2.4 服务依赖关系图（哪个组件依赖哪个，启动顺序）
   2.5 有状态/无状态组件标注（影响将来水平扩展能力）

3. 权限体系
   3.1 权限矩阵表（角色×功能×数据范围）
   3.2 数据过滤规则（每级角色的查询过滤逻辑）
   3.3 关键权限逻辑伪代码
   3.4 前端路由守卫规则（哪些路由需要什么角色才能访问）
   3.5 敏感操作二次确认清单（哪些操作需要输入密码/验证码确认）

4. 数据模型
   4.1 ER关系图（ASCII）
   4.2 每张表的完整CREATE TABLE DDL
   4.3 JSON/JSONB字段的完整结构示例
   4.4 索引清单（CREATE INDEX语句）
   4.5 数据库迁移策略（使用什么工具，如Alembic/Flyway/Prisma Migrate）
   4.6 大表迁移注意事项（超过百万行的表做ALTER TABLE可能锁表，需用在线DDL工具）
   4.7 数据增长预估与归档策略（哪些表增长最快？超过N个月的数据是否归档？）

5. 功能模块详细说明
   5.N 每个模块包含：
       - 功能描述
       - 业务规则（边界条件+特殊逻辑）
       - 用户交互流程（正常路径）
       - 异常处理（失败/超时/无数据/无权限时用户看到什么）
       - 异步任务说明（如有：触发方式、队列配置、超时处理、重试策略）
       - 数据流向（前端调哪个API→哪个Service→读写哪张表→返回什么）
       - 幂等性设计（关键写入接口如何防重复提交）
       - 并发控制（多人同时操作同一数据时的锁策略：乐观锁version字段/悲观锁）

6. AI Prompt 设计（如涉及AI）
   6.N 每个Prompt包含：
       - 完整System Prompt文本
       - 输入变量说明
       - 输出格式定义（JSON Schema）
       - Token控制策略
       - 降级方案（API不可用时怎么办）
       - AI返回格式异常时的容错处理（JSON解析失败怎么办？字段缺失怎么办？）
       - Fallback值定义（AI拿不到结果时前端展示什么）

7. 页面清单与路由
   7.1 按角色分组的页面表（页面名、路由、说明）
   7.2 列表页规范（排序、分页、搜索、筛选）
   7.3 核心用户流程图（每个角色）
   7.4 前端路由守卫逻辑（未登录跳登录页、角色不匹配跳403、首次改密拦截）
   7.5 前端弱网/离线兜底方案（长表单暂存、提交失败重试、断网提示）

8. API 接口清单
   8.N 按模块分组的API表（方法、路径、说明、角色）
   8.X 每个API的请求体/响应体Schema定义（字段名、类型、必填/可选、校验规则）
   8.Y 关键API的请求/响应JSON示例（至少覆盖：登录、核心业务提交、列表查询）
   最后标注API总数

9. 高并发/性能方案
   9.1 峰值流量估算（用公式计算，不拍脑袋）
   9.2 并发承载分析表（每个操作的同时人数、处理方式、响应时间、结论）
   9.3 缓存策略（什么数据缓存、TTL多久、缓存失效策略）
   9.4 缓存防护策略（穿透→缓存空值/布隆过滤器；击穿→互斥锁；雪崩→TTL加随机偏移）
   9.5 异步队列配置（Worker数量、并发数、超时时间、重试策略）
   9.6 数据库连接池配置
   9.7 限流规则（全局/单用户/单IP）
   9.8 乐观锁/悲观锁使用场景说明
   9.9 幂等性设计方案（哪些接口需要、用什么机制实现）

10. 安全设计方案
    10.1 五层防护（网络/应用/数据/运维/密钥生命周期）
    10.2 每层的具体措施和实现方式
    10.3 API错误码定义
    10.4 日志设计规范（结构化格式、链路追踪、分级标准、脱敏规则）
    10.5 敏感数据加密存储方案（如涉及PII数据）

11. 移动端适配策略（如需要）
    - 断点定义
    - 关键页面的PC/手机布局差异
    - 浏览器兼容要求

12. 开发规约与已知陷阱（v4.1新增）
    **本章是给Claude Code的"踩坑预警"，动手写代码前必读。**

    12.1 类型统一规约
        - 时间类型：所有时间字段统一用 [TIMESTAMPTZ / TIMESTAMP]，代码中统一用 [naive UTC / aware UTC]，禁止混用
        - 主键类型：所有表统一用 [Integer自增 / UUID]，不混用
        - 空值语义：数据库NULL、Python None、JSON null、前端空字符串""的处理规则
        - 枚举值：角色/状态等枚举统一小写（或统一大写），前后端一致

    12.2 框架组合兼容性验证
        - 列出本项目技术栈中已知的版本兼容陷阱
        - 示例（按实际技术栈填写）：
          · pytest-asyncio ≥1.0 必须显式设置 asyncio_mode="auto"，event loop scope 默认为 function
          · SQLAlchemy 2.0 async 模式下禁止 lazy loading，所有关联查询必须用 selectinload()/joinedload()
          · Pydantic v2 的 orm_mode 改为 from_attributes，validator 改为 field_validator
          · Alembic 异步模式需要特殊 env.py 配置
        - **开发第一天必须先跑一个最小可运行测试（smoke test）验证框架组合可用**

    12.3 外键依赖与数据操作顺序
        - 表的拓扑排序图（按FK依赖关系）
        - 种子数据插入顺序（按依赖正序）
        - 测试数据清理顺序（按依赖逆序）
        - 每张表的级联删除策略：CASCADE / SET NULL / RESTRICT，标注理由
        - 示例：删除用户时，审计日志CASCADE（跟着删）、订单SET NULL（保留记录但清除关联）、正在进行的任务RESTRICT（禁止删除）

    12.4 全局副作用清单与测试隔离
        - 列出所有有全局副作用的组件：
          · Redis缓存（哪些key、TTL多久、测试后是否清理）
          · 全局变量/单例（如配置缓存、连接池）
          · 文件系统（上传目录、临时文件）
          · 环境变量（测试中是否修改、修改后是否恢复）
        - conftest.py 必须包含的 autouse fixture：
          · 数据库事务回滚或按FK逆序清理
          · Redis FLUSHDB（测试用独立db编号）
          · 临时文件清理
        - **这些fixture必须在写第一个测试文件之前就设计好**

    12.5 本地与生产环境差异对照
        | 维度 | 本地开发 | 生产环境 | 常见踩坑 |
        |------|---------|---------|---------|
        | 数据库版本 | （标注） | （标注） | 函数/语法差异 |
        | Python/Node版本 | （标注） | （标注） | 语法兼容性 |
        | 操作系统 | macOS/Windows | Linux | 文件路径大小写敏感 |
        | 时区 | 本地时区 | UTC | datetime.now()差8小时 |
        | 文件路径 | 绝对路径可能不同 | 容器内路径 | 路径硬编码会报错 |
        | 网络 | 前后端同端口 | 不同域名 | CORS/HTTPS混合内容 |
        | 数据量 | 几十条 | 几十万条 | 慢查询/内存溢出 |
        | 连接池 | 默认5 | 需按并发调整 | 连接池耗尽 |

    12.6 数据库连接与ORM注意事项
        - 连接池配置：pool_size / max_overflow / pool_pre_ping=True（自动检测断连）
        - 异步ORM：禁止lazy loading，必须显式声明加载策略
        - 事务边界：哪些操作需要在同一事务内？事务失败时的回滚行为？
        - Migration注意：多人/多Session开发时可能出现多head，合并策略是什么？

13. 测试策略
    13.1 测试基础设施验证（v4.1新增）
        - **开发第一天必须完成**：写一个最小化的异步测试用例，验证 pytest + 异步框架 + 数据库 + Redis 的组合能正常运行
        - 测试数据库：使用独立的测试数据库（不复用开发库）
        - 测试Redis：使用独立的db编号（如db=1用于测试，db=0用于开发）
    13.2 测试数据管理
        - conftest.py 的 autouse fixture 设计：数据清理策略（事务回滚 vs 按FK逆序DELETE）
        - 测试数据工厂（factory）：如何快速创建测试用的用户/订单等数据
        - 全局副作用隔离：Redis清理、文件清理、环境变量恢复
    13.3 后端测试：核心API的测试用例描述（至少覆盖登录、权限校验、核心业务流程、幂等性验证）
    13.4 前端测试：关键页面的验收标准
    13.5 压力测试：场景定义、工具选择、通过标准

14. CI/CD 与部署方案
    14.1 仓库结构（Monorepo/多仓库）
    14.2 Docker Compose完整配置（所有服务的定义）
    14.3 自动部署流程（GitHub Actions / 手动部署步骤）
    14.4 本地开发环境搭建命令
    14.5 多环境管理（Dev/Staging/Prod环境差异对照表）

15. 架构韧性设计（v4.0新增）
    18.1 单点故障识别表（每个组件是否单点、V2消除方案）
    14.2 故障影响矩阵（每个组件故障时哪些功能受影响）
    14.3 RTO/RPO定义（恢复时间目标、恢复点目标）
    14.4 熔断器设计（外部API调用的熔断参数和降级行为）
    14.5 自动恢复机制（进程重启、连接重连、磁盘告警）
    14.6 优雅降级层级定义（L0正常→L1部分降级→L2核心降级→L3紧急降级→L4完全不可用）

16. 跨系统状态一致性（v4.0新增，如涉及支付/外部交易）
```

## 写作标准

### 数据模型
- 完整CREATE TABLE SQL（字段类型+约束+默认值）
- 外键关系明确
- JSON字段给完整结构示例
- 索引给CREATE INDEX语句
- 标注使用什么迁移工具（Alembic/Flyway/Prisma Migrate等）
- **乐观锁字段**：需要并发控制的表标注version字段
- **数据增长预估**：每张核心表标注预计月增长量，超过50万行的表标注归档策略

### API接口
- HTTP方法 + 完整路径 + 说明 + 允许角色
- 每个API必须定义请求体和响应体的Schema（字段名、类型、必填/可选、校验规则）
- 关键API必须给出请求体和响应体的JSON示例
- 按模块分组，最后统计总数
- 分页接口标注默认值（page_size, sort_order）
- 路径命名统一规范：资源名复数（/students）、风格一致、层级清晰
- **幂等接口标注**：哪些接口需要幂等，用什么机制（幂等键/唯一约束/乐观锁）
- **超时阈值标注**：每个API的预期响应时间和超时阈值

### 权限
- 角色×功能×数据范围的完整交叉表
- 数据过滤逻辑给伪代码
- 前端路由守卫规则：哪些路由对应什么角色
- **敏感操作清单**：哪些操作需要二次确认

### 安全
- 每种攻击类型逐一说明防护方式
- 密码加密具体到算法+参数
- CORS/安全头部/上传校验给具体配置
- 不能写"注意安全"这种空话
- **日志规范**：结构化格式、链路追踪、分级标准、脱敏规则
- **敏感数据加密**：需要字段级加密的数据列出加密方案

### 目录结构
- 列到文件级别
- 关键文件标注用途
- 包含测试目录、配置文件、Docker文件、CI/CD文件

### 环境变量
- 完整.env模板+注释
- 敏感变量标注"不入代码库"
- 提供.env.example
- **多环境差异标注**：哪些变量在Dev/Staging/Prod不同

### 第三方依赖
- 锁定版本号（如 fastapi==0.109.0，不要写 fastapi>=0.109）
- 特殊依赖标注安装方式（如 torch CPU版: torch==2.1.0+cpu）
- **EOL关注**：标注主要运行时和依赖库的安全支持截止时间

### Docker Compose
- 完整的服务定义（Web/DB/Redis/Worker等）
- healthcheck配置
- 数据卷持久化
- 环境变量引用.env
