# 上线准备检查清单（Go-Live Checklist）— v4.0 新增

PRD标记为Final后，生成此清单作为附录。上线前逐项打勾确认。

## 一、基础设施确认（8项）

- [ ] DNS解析是否生效？（dig命令验证）
- [ ] SSL证书是否正确安装？（浏览器无警告、HTTPS强制跳转正常）
- [ ] 服务器防火墙规则是否正确？（仅开放80/443/SSH）
- [ ] 数据库种子数据是否已导入？（超管账号、初始配置、字典数据）
- [ ] 环境变量是否都配齐？（逐一比对.env.example）
- [ ] 定时任务（cron/celery beat）是否启动？
- [ ] 备份任务是否运行并验证过？（手动触发一次备份+恢复验证）
- [ ] 监控告警是否配置并测试过？（发一条测试告警确认能收到）

## 二、第三方服务预热（6项）

- [ ] **支付商户号**：是否已有小额交易记录？新商户号需提前用小额交易"养号"（至少1-2周，逐步提升日交易额），否则瞬时高并发会触发平台风控锁定
- [ ] **短信/邮件服务**：新账号日发送量上限是多少？是否需要提前申请提额？已申请的额度是否足够首批用户量？
- [ ] **AI API**：新账号的TPM/RPM限制是多少？是否需要提前申请提升配额？当前配额是否能支撑预估峰值？
- [ ] **云存储（OSS/S3）**：Bucket是否创建？CORS是否配置？CDN是否开启？
- [ ] **域名备案**（国内部署）：ICP备案是否已通过？（未备案会被运营商拦截）
- [ ] **DNS TTL**：上线前是否已将TTL降低至60-300秒？（方便紧急情况下快速切换IP）

## 三、灰度发布策略（5项）

- [ ] **灰度阶段定义**：
  1. 内部测试（团队5-10人）→ 验证核心功能
  2. 小范围公测（10%用户 或 指定用户群）→ 观察1-3天
  3. 扩大范围（50%用户）→ 观察1天
  4. 全量上线
- [ ] **灰度期间监控指标**：错误率、API响应时间P95、支付成功率、第三方API调用成功率、服务器CPU/内存/磁盘
- [ ] **灰度回退条件**：
  - 错误率 > 5% → 立即回退
  - 核心API P95响应时间 > 3秒 → 立即回退
  - 支付成功率 < 95% → 立即回退
  - 任何核心功能不可用 → 立即回退
- [ ] **灰度放量机制**：通过什么方式控制灰度比例？（用户白名单/按注册时间分桶/特性开关）
- [ ] **灰度期间的用户反馈渠道**：是否准备好收集灰度用户反馈的方式？

## 四、上线后 Smoke Test（5项）

上线后前5分钟必须手动验证：

- [ ] 能正常打开首页（HTTPS、无报错）
- [ ] 能正常登录（各角色至少测一个）
- [ ] 核心业务功能跑通一次（从头到尾完整走一遍）
- [ ] 支付/外部API能调通（如有）
- [ ] 移动端打开正常（如需要移动端适配）

## 五、快速回滚方案（4项）

- [ ] **回滚触发条件**（明确写出，不要犹豫）：
  - 任何P0级事故 → 立即回滚
  - Smoke Test任何一项不通过 → 立即回滚
  - 上线后30分钟内错误率持续>5% → 回滚
- [ ] **回滚步骤**：具体到执行哪几条命令（docker compose切换到上一版本镜像？git回退+重新构建？）
- [ ] **回滚后验证**：回滚后需要重新跑一遍Smoke Test
- [ ] **数据库回滚**：如果本次上线包含数据库迁移，回滚时数据库怎么处理？（有migration rollback脚本？还是不可回滚？）

## 六、人工应急准备（4项）

- [ ] **应急联系人清单**是否就绪？（值班运维、技术负责人、业务负责人、第三方服务商支持电话/工单入口）
- [ ] **用户沟通模板**是否准备好？（系统维护公告、故障恢复公告、退款/补偿通知）
- [ ] **通知渠道**是否确定？（通过什么方式通知用户？站内公告？微信群？短信？邮件？）
- [ ] **线下兜底方案**是否可执行？（如果系统完全不可用，业务是否有线下替代方案？）

## 七、上线时间窗口（3项）

- [ ] **避免高风险时间**：不在周五下午/节假日前/业务高峰期上线
- [ ] **推荐上线时间**：业务低谷期（如工作日上午10点），确保后续4-6小时有人值守
- [ ] **值守安排**：上线后谁值守？值守多长时间？（建议至少值守到第一个业务高峰期过去）

## 八、上线后持续关注（5项）

上线后第一周每天检查：

- [ ] 错误日志有无异常增长
- [ ] 数据库慢查询是否出现
- [ ] 磁盘空间使用趋势（日志/上传文件增长速度）
- [ ] 内存使用趋势（是否有泄漏迹象——内存持续增长不释放）
- [ ] 第三方API调用量/费用是否在预期范围内

---

## 使用方式

1. PRD完成阶段五（扫雷审查）后，根据项目情况生成本清单
2. 开发完成后，上线前逐项打勾确认
3. 有任何一项为"否"或"未确认"，不建议上线
4. 本清单作为PRD附录的一部分交付
